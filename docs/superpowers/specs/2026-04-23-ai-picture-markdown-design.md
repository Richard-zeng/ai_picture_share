# AI 图片作品展示页设计文档

## 1. 项目目标

为用户创建一个适合长期维护的 GitHub Markdown 展示页方案，用来分享不同提示词生成的图片作品，并配套一份从 0 开始的 GitHub 上传教程。

本次设计聚焦 3 个交付物：

- 一个可直接复用的 Markdown 模板
- 一个带示例的填写方式
- 一份面向新手的 GitHub 上传与 `push` 教程

## 2. 用户背景与约束

用户当前特征：

- 没有 GitHub 使用经验
- 不熟悉 Git 和 `push` 流程
- 图片主要保存在本地
- 希望先发布为单独的 `.md` 文件，而不是仓库首页 `README.md`
- 希望页面主要使用中文
- 希望这个页面可以长期持续更新

当前环境约束：

- 当前工作区尚未确认存在 Git 仓库
- 当前终端环境中 `git` 命令不可用，说明 Git 尚未安装或未加入 PATH

## 3. 设计结论

### 3.1 展示形式

采用“案例卡片作品页”风格，而不是纯海报式或纯图库式。

选择原因：

- 比纯大图海报更适合长期更新
- 比纯图库更适合记录提示词与模型信息
- 在 GitHub Markdown 场景下结构清晰，学习成本低

### 3.2 发布位置

使用单独的 Markdown 文件发布，不直接覆盖仓库首页。

建议文件名：

- `AI-Prompt-Showcase.md`

这样做的优点：

- 不影响未来仓库首页的其他用途
- 更适合先练习 GitHub 内容组织
- 后续可以再决定是否把精选内容同步到 `README.md`

### 3.3 内容粒度

每组案例采用“标准型”记录字段：

- 标题
- 图片
- 提示词
- 使用模型
- 简短说明

不纳入本次默认模板的字段：

- 复杂参数表
- 长篇心得
- 正反向提示词分栏

这些内容暂时不放入默认模板，以降低长期维护成本；如后续需要，可增补进阶版模板。

## 4. 文件组织设计

建议使用如下目录结构：

```text
AIpicture/
├─ AI-Prompt-Showcase.md
└─ images/
   ├─ case-01/
   │  ├─ image-01.png
   │  └─ image-02.png
   ├─ case-02/
   │  └─ image-01.png
   └─ case-03/
      └─ image-01.png
```

设计原则：

- Markdown 文件和图片一起放进同一个 GitHub 仓库
- 通过相对路径引用图片，确保 GitHub 页面正常渲染
- 每组案例一个独立文件夹，便于长期维护和扩展

## 5. 页面结构设计

目标页面结构如下：

### 5.1 顶部说明区

用于快速告诉读者这个页面是什么，建议包括：

- 页面标题
- 一句简介
- 内容说明
- 更新方式说明

### 5.2 案例目录区

在文档前部放一个简短目录，方便别人快速跳转到具体案例。

### 5.3 案例卡片区

每组案例作为一个独立板块，按以下顺序展示：

1. 案例标题
2. 图片预览
3. 提示词
4. 使用模型
5. 简短说明

### 5.4 页尾说明区

建议放：

- 持续更新说明
- 使用声明
- 联系方式或主页链接（可选）

## 6. 模板风格要求

模板应满足以下要求：

- 中文为主
- 开箱即用，复制后稍作修改即可发布
- 视觉上简洁清晰，适合 GitHub 原生 Markdown 渲染
- 不依赖复杂 HTML 和外部前端资源
- 支持用户后续手动追加新案例

为了兼顾美观和稳定性，模板可少量使用：

- Markdown 标题层级
- 引用块
- 代码块
- 分割线
- 简单的图片标签或 Markdown 图片语法

## 7. GitHub 教学设计

教程必须面向完全没有基础的用户，覆盖完整流程。

教程范围包括：

1. 注册 GitHub 账号
2. 创建新的仓库
3. 在 Windows 上安装 Git
4. 配置 Git 用户名和邮箱
5. 将 Markdown 文件和图片整理到本地文件夹
6. 初始化仓库（如果需要）
7. 连接远程 GitHub 仓库
8. 执行 `add`、`commit`、`push`
9. 在 GitHub 网页上确认文件和图片是否显示正常

教程表达要求：

- 每一步都尽量写清楚按钮位置或命令用途
- 不假设用户已经理解命令行
- 对关键概念做短解释，例如“仓库”“commit”“push”分别是什么意思
- 尽量减少一次性的信息量，避免让新手焦虑

## 8. 示例内容设计

除模板外，还需要提供一个可直接参照的填写示例，用来帮助用户理解：

- 图片应该放在哪个文件夹
- Markdown 里图片路径应该怎么写
- 每组案例的标题和说明怎么组织

示例内容不需要依赖真实线上图片链接，优先使用本地相对路径示例。

## 9. 成功标准

如果本次交付达成以下结果，则视为成功：

- 用户拿到后能直接照着填写自己的案例
- 用户理解本地图片如何在 GitHub Markdown 中显示
- 用户知道如何从本地文件夹上传到 GitHub
- 用户即使此前没有 GitHub 经验，也能按教程完成第一次 `push`

## 10. 风险与应对

### 风险 1

用户电脑未安装 Git，导致命令无法执行。

应对：

- 在教程中单独加入 Git 安装与验证步骤

### 风险 2

用户把图片放在错误目录，导致 GitHub 页面图片无法显示。

应对：

- 在模板和示例中都明确展示相对路径写法

### 风险 3

用户第一次接触 `commit`、`push` 时概念混乱。

应对：

- 在教程中加入一句话解释每个核心命令的作用

## 11. 下一步实施范围

在用户确认本设计后，进入实施阶段，产出以下内容：

- 最终版 Markdown 模板
- 示例版 Markdown 文件片段
- GitHub 上传教程

本阶段不做的事情：

- 直接代替用户注册 GitHub
- 直接替用户在线创建仓库
- 直接替用户执行真实的远程 `push`

原因是当前环境尚未具备 Git 配置与 GitHub 账号授权条件，且这部分需要用户本人操作或登录。
